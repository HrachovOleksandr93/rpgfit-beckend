## 1. Architecture — Extended `Mob` entity

### 1.1 New fields

| Field | Type | Nullable | Default | Notes |
|-------|------|----------|---------|-------|
| `realm` | enum | no | `neutral` | `olympus \| asgard \| dharma \| duat \| nav \| shiba \| neutral` |
| `class_tier` | enum | no | `I` | `I \| II \| III \| IV` (from docx Part III.1) |
| `behavior` | enum | no | `physical` | `physical \| ritual \| oracle_task \| team` |
| `archetype` | enum | no | `beast` | `humanoid \| beast \| spirit \| undead \| construct \| divine \| chimera \| swarm` |
| `visual_keywords` | json/array<string> | no | `[]` | 3–5 short descriptors for artists |
| `is_champion` | bool | no | `false` | marks the variant record (see `mobs-champion-variants.md`) |
| `champion_decoration` | string | yes | `null` | e.g. `"apple_watch"`, `"powerbank_necklace"` — set only if `is_champion=true` |
| `accepts_champion` | bool | no | `true` | `false` for ethereal/swarm/incorporeal archetypes — excluded from champion spawn pool |

### 1.2 Rarity ↔ class_tier mapping (working rule)

| Rarity | Class tier | Expected % of realm roster |
|--------|-----------|---------------------------|
| common | I | 35% |
| uncommon | I | 25% |
| rare | II | 25% |
| epic | III | 12% |
| legendary | IV | 3% |

Mapping is a **default** — a rare mob can manually be raised to Class III if lore requires (e.g. a mid-tier Aesir vs. a Titan). Command `app:seed-mobs` validates that every legendary is tier IV.

### 1.3 Behavior enum — decision rules

- `physical` → any mob that expects raw damage from a workout session (default for 70% of roster).
- `oracle_task` → mob requires completing a structured physical challenge (e.g. "20 pull-ups this session"). Tier I–II only.
- `ritual` → mob damage gated by knowledge/artifact/specific workout type (yoga mats for Dharma spirits; cold-shower for Nav Vodyanyk).
- `team` → mob only spawnable in raid mode; Class III+ default. Enforced by `BattleService` when mode ≠ raid.

### 1.4 `accepts_champion` decision matrix

- **Accepts (`true`):** humanoid, beast, undead (body-bound), construct, chimera. Can plausibly wear/carry an Apple Watch, powerbank, gym-weight.
- **Rejects (`false`):** spirit (incorporeal), swarm (collective), divine (tier IV Olympians / Aesir / first-tier Devas — dignity protection per founder tone note), some ethereal Nav entities.

### 1.5 HP / XP formula — UNCHANGED

Per `BUSINESS_LOGIC.md §10`:
- `HP = 20 · level^1.5 + 40 ± 20%`
- `XP = (4.2·level² + 28·level) / 15 · rarity_multiplier ± 10%`

New fields are **metadata only**. `realm` + `class_tier` do **not** affect base HP/XP — they affect artifact damage multipliers (see `docs/vision/mobs.md`: realm-bound artifact +40%), spawn filtering, and visual/flavor. No leveling refactor required.

## 2. YAML generator design (replaces CSV)

### 2.1 File layout

```
rpgfit-beckend/data/mobs/
  olympus.yaml
  asgard.yaml
  dharma.yaml
  duat.yaml
  nav.yaml
  shiba.yaml
  neutral.yaml      # legacy backfill, existing 2000 CSV rows migrated here
```

Each file: array of mob definitions (one entry = one mob, not one-per-level). Level variants generated by seeder from `level_range: [min, max, step]` → produces `(max-min)/step + 1` rows in DB with formula-applied HP/XP.

### 2.2 Entry schema (conceptual, not emitted)

Each YAML entry holds: `slug` (unique), `name_ua`, `name_en`, `rarity`, `realm`, `class_tier`, `behavior`, `archetype`, `visual_keywords` (array), `accepts_champion`, `flavor_ua`, `flavor_en`, `level_range`, `source_url` (optional — for specialized creatures; canonical ones reference realm-level source).

### 2.3 Command

- **New:** `app:seed-mobs` — reads all `data/mobs/*.yaml`, upserts by slug, expands level-range into DB rows using existing HP/XP formulas.
- **Deprecated:** `app:import-mobs` (CSV). Kept for one release as read-only fallback; removed in Phase 5 of migration.

### 2.4 Champion spawning

Not a YAML concern. Champion variants generated at runtime in `BattleService` / new `MobSelectionService`:
- When a mob is selected, roll 10–15% chance for champion variant.
- Filter requires `accepts_champion = true`.
- Champion variant picks a `champion_decoration` from static pool defined in `docs/vision/mobs-champion-variants.md` (apple_watch, powerbank, gym-weight, smartphone, hydration_pack, …).
- Champion gets +10% HP, +25% XP, unique loot-drop bias toward cross-world items.

## 5. Migration plan (high-level, 6 phases)

- **P1 — Schema.** Doctrine migration adds 8 new `Mob` columns (per §1.1). Backfill all existing rows to `realm=neutral, class_tier=I, behavior=physical, archetype=beast, accepts_champion=true`. No data loss, no HP/XP change.
- **P2 — Pilot seeder.** Ship `olympus.yaml` (the 40 above) + new `app:seed-mobs` command. Keep `app:import-mobs` CSV command live. Feature-flag in `MobSelectionService`: Olympus mobs only spawn if `feature.realm_bestiary_olympus = on`.
- **P3 — Champion spawn.** Add `MobSelectionService::maybeAsChampion()` — 10–15% roll, respects `accepts_champion`, attaches `champion_decoration` + loot-bias. UI badge for champions in battle screen.
- **P4 — Realm rollout.** Expand seeds from §4 into full 40-mob YAML files for Asgard, Dharma, Duat, Nav, Shiba (5 realms × 40 = 200 new mobs). Likely 1 content-agent run per realm.
- **P5 — Legacy CSV retire.** Convert `data/mobs.csv` (2000 rows) into `neutral.yaml` via one-off migration script with auto-mapping (level→tier default, no realm). Deprecate `app:import-mobs`; remove in next release.
- **P6 — Artifact coupling.** Wire realm-bound artifact damage multiplier (+40% per `BUSINESS_LOGIC.md` addition): `if mob.realm === artifact.realm → damage × 1.4`. Update `BattleResultCalculator`. Documented in `BUSINESS_LOGIC.md §10`.
